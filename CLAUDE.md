# プロジェクトメモリ

## プロジェクト概要

**Rust製 Git GUI クライアント** — 軽量でシンプルな、自分用 Git GUI ツール。

- **動機**: 既存ツール（SourceTree・GitKraken 等）は重くて複雑すぎる。VSCode ライトモード風のシンプルなデザインで、自分が欲しいものだけを作る。
- **デザインリファレンス**: VSCode 拡張「Git Graph」— コミット一覧 + ブランチグラフ + diff 表示。
- **提供形態**: Windows ネイティブアプリ（自分用ビルド。配布は未定）。
- **MVP の核**: 「読むだけ」から始める。コミット履歴一覧 + diff 確認の 2 ペイン。

詳細は `docs/product-requirements.md` 以下の永続ドキュメントを正とする。

## 技術スタック

- **言語**: Rust（開発者はRust未経験。Claude Codeと協力して実装）
- **UI フレームワーク**: egui（純Rust製の即時モードGUI。軽量・クロスプラットフォーム）
- **Git バックエンド**: git2 クレート（libgit2 の Rust バインディング。高速・プロセス起動なし）
- **ビルドツール**: Cargo
- **プラットフォーム**: Windows のみ（当面）

### レイヤー構成

```
UI（egui ウィジェット）
  ↓
ViewModel / App State（Rust struct）
  ↓
Git ロジック（git2 クレート）
```

逆方向・横断の依存は禁止。

## スペック駆動開発の基本原則

### 基本フロー

1. **ドキュメント作成**: 永続ドキュメント(`docs/`)で「何を作るか」を定義
2. **作業計画**: ステアリングファイル(`.steering/`)で「今回何をするか」を計画
3. **実装**: tasklist.mdに従って実装し、進捗を随時更新
4. **検証**: テストと動作確認
5. **更新**: 必要に応じてドキュメント更新

### 重要なルール

#### ドキュメント作成時

**1ファイルずつ作成し、必ずユーザーの承認を得てから次に進む**

承認待ちの際は、明確に伝える:
```
「[ドキュメント名]の作成が完了しました。内容を確認してください。
承認いただけたら次のドキュメントに進みます。」
```

#### 実装前の確認

新しい実装を始める前に、必ず以下を確認:

1. CLAUDE.mdを読む
2. 関連する永続ドキュメント(`docs/`)を読む
3. Grepで既存の類似実装を検索
4. 既存パターンを理解してから実装開始

#### ステアリングファイル管理

作業ごとに `.steering/[YYYYMMDD]-[タスク名]/` を作成:

- `requirements.md`: 今回の要求内容
- `design.md`: 実装アプローチ
- `tasklist.md`: 具体的なタスクリスト
- `retrospective.md`: 実装後の振り返り（モード3で作成。tasklist.md とは別ファイル）

命名規則: `20260702-add-commit-history` 形式（`[YYYYMMDD]-[タスク名]`）

**軽微な作業の振り返り**: ステアリングを切るほどでもないバグ修正・小改善でも、残すべき
学び・申し送り（再発防止、ハマりどころ、設計判断、次回への注意）があれば、`retrospective.md`
**だけ**を `.steering/[YYYYMMDD]-[作業名]/` に残す（steeringスキルのモード3・ケースB）。
学びが何もない軽微作業（typo修正など）は作成不要。

#### ステアリングファイルの管理

**作業計画・実装・検証時は`steering`スキルを使用してください。**

- **作業計画時**: `Skill('steering')`でモード1(ステアリングファイル作成)
- **実装時**: `Skill('steering')`でモード2(実装とtasklist.md更新管理)
- **検証時**: `Skill('steering')`でモード3(振り返り)

詳細な手順と更新管理のルールはsteeringスキル内に定義されています。

## ディレクトリ構造

### 永続的ドキュメント(`docs/`)

アプリケーション全体の「何を作るか」「どう作るか」を定義:

#### 下書き・アイデア（`docs/ideas/`）
- 壁打ち・ブレインストーミングの成果物
- 技術調査メモ
- 自由形式（構造化は最小限）

#### 正式版ドキュメント

- **product-requirements.md** - プロダクト要求定義書
- **functional-design.md** - 機能設計書
- **architecture.md** - 技術仕様書
- **repository-structure.md** - リポジトリ構造定義書
- **development-guidelines.md** - 開発ガイドライン
- **glossary.md** - ユビキタス言語定義(用語集)

### 作業単位のドキュメント(`.steering/`)

特定の開発作業における「今回何をするか」を定義:

- `requirements.md`: 今回の作業の要求内容
- `design.md`: 変更内容の設計
- `tasklist.md`: タスクリスト
- `retrospective.md`: 実装後の振り返り（モード3で作成）

### .claude カタログの維持

`.claude/` 配下の command / skill / agent は `.claude/README.md` に目次化している。
**command / skill / agent を追加・削除・リネームしたら、`.claude/README.md` も必ず同じ変更で更新する**こと。

## Git運用ルール(ブランチ戦略)

**実装・修正作業は必ず `feature` ブランチで行う。`main` への直接コミットは禁止。**

### ワークフロー

1. **作業開始時**: 先に `main` を `git pull` で最新化し、`git worktree` で専用作業ディレクトリを作成
   - 命名規則: `feature/<説明>` (例: `feature/commit-history`)
   - 作成例: `git worktree add ../git-client-<説明> -b feature/<説明> main`

2. **作業完了時**:
   1. コミット前に振り返り(`retrospective.md`)を作成
   2. `main` を pull して最新化
   3. `feature` ブランチに最新 `main` を取り込む
   4. コミット前に**クルトワ(security-engineer)のセキュリティレビュー**を実施
   5. 変更一式をコミット → push → PR 作成(`gh pr create`)
   6. `main` へマージ（**Merge commit**方式）

3. **マージ完了後**: ブランチ・worktree を両方削除

### 補足

- 軽微な修正でも原則 `feature` ブランチを切る
- `git worktree` の利用は任意ではなく**必須**

## コミット前セキュリティレビュー

**コミットを実行する前に、必ず security-engineer によるセキュリティレビューを実施すること。**

### チェック項目

- **シークレット/キーのハードコーディング**: API キー、トークン等がソースに埋め込まれていないか
- **コマンドインジェクション**: シェル呼び出しがある場合、ユーザー入力を適切にエスケープしているか
- **パスの安全性**: ユーザーが指定するリポジトリパスを適切に検証しているか

## セッション引き継ぎ時の作業原則

- サマリーを信頼しない。修正済みと記載されていても、**実際のファイルをReadツールで確認**してから進む
- セッション引き継ぎ後に変更対象ファイルがある場合、まず対象ファイルを読んで現在の状態を把握する
