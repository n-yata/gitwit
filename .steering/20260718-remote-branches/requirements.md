# 要求内容

## 概要
ブランチ切り替えドロップダウンに、実際に `git fetch` を実行して取得したリモートブランチ
（`origin/xxx` 等）の一覧を非同期（UIをブロックしない形）で表示する。

## 背景
`.steering/20260718-branch-switch/` で実装したブランチ切り替え機能は、ローカルブランチのみを
対象にし、リモートブランチの一覧・チェックアウトは明示的にスコープ外としていた。今回、ユーザーから
「リモートブランチの一覧も、非同期で取得できたら一覧に出すようにして」という追加要求があった。
ヒアリングの結果、ローカルにキャッシュ済みの `refs/remotes` を読むだけでなく、実際に `git fetch`
を実行して最新のリモートブランチ情報を取得することが目的と確認した。

gitwit にとって初めてのネットワークI/O・初めてのバックグラウンドスレッド導入となるため、
architecture-designer（バルベルデ）による設計を経て方針を固めた
（`C:\Users\yata1\.claude\plans\lucky-puzzling-storm-agent-a3481661e3267f3d8.md`）。

## 実装対象の機能

### 1. リモートブランチの非同期取得
- ツールバーに「リモート取得」ボタンを追加する。
- クリックすると、バックグラウンドスレッドで全リモートに対して `git fetch` を実行する。
  UIスレッドはブロックされず、fetch 中も他の操作（コミット一覧の閲覧等）が可能。
- fetch 中はボタンを無効化し、進行中であることが分かる表示（スピナー等）を出す。
- fetch 完了後、リモートブランチ一覧（例: `origin/main`）をブランチドロップダウンに反映する。

### 2. 認証
- 認証は SSH エージェント / Git の既存認証（credential helper, Windows 資格情報マネージャー）に
  委譲する。アプリ側でパスワード等のシークレットを保持・入力させることはしない。

### 3. リモートブランチの表示
- ブランチ選択ドロップダウンに、ローカルブランチとは区別できる形でリモートブランチを追記表示する。
- リモートブランチの選択によるチェックアウト（detached HEAD やトラッキングブランチ作成）は対象外。
  今回は「一覧に出す」までがスコープ。

## 受け入れ条件

### リモートブランチの非同期取得
- [ ] ツールバーに「リモート取得」ボタンが表示される
- [ ] ボタンをクリックすると、UIがフリーズせずコミット一覧のスクロール等の他の操作を継続できる
- [ ] fetch 中はボタンが無効化され、進行中であることが視覚的に分かる
- [ ] fetch 成功後、ドロップダウンにリモートブランチ（`origin/xxx` 形式）が表示される
- [ ] ボタンを連打しても多重に fetch が起動しない

### 認証
- [ ] SSH エージェント等、システムの既存認証で fetch が成功する（追加のログイン入力UIは不要）

### リモートブランチの表示
- [ ] ドロップダウン上でローカルブランチとリモートブランチが区別できる
- [ ] リモートブランチをクリックしてもチェックアウトは行われない（一覧表示のみ）

### エラー処理
- [ ] ネットワーク未接続・認証失敗などで fetch が失敗しても、アプリはクラッシュしない
- [ ] fetch 失敗時、ボタンは再度クリック可能な状態に戻る

## 成功指標
- 「リモート取得」ボタン操作中もアプリの操作性（コミット一覧の閲覧・スクロール等）が損なわれない

## スコープ外

以下はこのフェーズでは実装しません:

- リモートブランチのチェックアウト（detached HEAD やローカルトラッキングブランチの作成）
- fetch のキャンセル機能
- 複数リモート（origin 以外）を意識した UI 上の出し分け（全リモートを対象に fetch はするが、
  UI 上は区別しない）
- push・pull・マージ等、fetch 以外のネットワーク操作

## 未決事項

- fetch 失敗時に `error_message` のエラーダイアログを出すか、UI上は静かに失敗させるかは、
  実装時に既存のエラーダイアログパターンを踏襲する（ダイアログ表示）方向で進める。過度に頻繁な
  ポップアップが体験を損なうと判断すれば、実装中に静かな失敗へ変更してもよい。
