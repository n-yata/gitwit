# 実装後の振り返り

## 作業概要

シャビの依頼で `stable-x86_64-pc-windows-msvc` Rustツールチェインを再インストールした。
作業中に `rustup toolchain uninstall` が途中で失敗し、ツールチェインが半壊状態
(マニフェスト消失、`cargo build`/`cargo run` 不能)になった。原因はVS Codeの
`rust-analyzer`(`rust-analyzer-proc-macro-srv`)がツールチェイン内の`rustc_driver-*.dll`を
ロードし続けていたこと。最終的に、ロックされたファイルを削除ではなくリネーム/移動する
ことで回避し、フルの再インストールに成功した(`rustc 1.96.1`)。

## 実装完了日

2026-07-04

## 計画と実績の差分

**計画と異なった点**:
- 当初は「アンインストール→インストール」の2コマンドで完結すると想定していたが、
  アンインストールの途中(ファイル削除のループ処理中)でロックされたDLLに当たって失敗し、
  ツールチェインのディレクトリがほぼ空(`bin/`に1ファイルのみ)という中途半端な状態で
  停止してしまった。この状態で `rustup toolchain install --force` を試しても、rustupが
  内部的に「変更なし」と誤判定して実際には再インストールしてくれず、余計に手間取った。
- 最終的な解決策は、ロックされたファイルを`Remove-Item`で削除するのではなく、
  `Rename-Item`→`Move-Item`で退避させてから`uninstall`→`install`をやり直す方法だった。
  Windowsでは「メモリにマップされた(実行中プロセスがロードした)DLL」は削除だけが
  拒否され、リネームや移動は通ることがある、という挙動を利用した。

**技術的理由でスキップしたタスク**: なし。

## 学んだこと

**技術的な学び**:
- `rustup toolchain uninstall`は「デフォルトツールチェインの削除」を警告つきで進めるが、
  削除処理はアトミックではなく、途中でロックに当たると**中途半端に破壊された状態**で
  停止する。実行前に、そのツールチェインを使っていそうなプロセス(`rust-analyzer`,
  `rust-analyzer-proc-macro-srv`等のIDE言語サーバ)がいないか確認しておくべきだった。
- Windowsのファイルロックは操作によって挙動が違う: ロード中のDLLは「削除」は拒否されるが
  「リネーム/移動」は通ることがある。削除が`Access denied`になった場合、即座に
  「アプリを閉じてもらう必要がある」と判断せず、まずリネーム/移動を試す価値がある。
- `rustup toolchain install --force`は、ディレクトリが壊れていても「rustcバージョンが
  読めない=既にインストール済み」と誤判定して実際には再インストールしないことがある。
  こうした場合は`uninstall`(を完全に成功させてから)→`install`のクリーンな手順を踏む
  必要がある。

**プロセス上の改善点**:
- 「ついでに直しておく」系のシステムレベルの破壊的操作(ツールチェイン再インストール等)は、
  完了までの手順を先に見積もり、途中で失敗した場合の環境が「元より悪化しないか」を
  考えてから着手すべきだった。今回は再インストール作業そのものが原因で、後続の別タスク
  (別ブランチのビルド検証)が一時的にブロックされる事態を招いた。

## 次回への改善提案
- 同様のツールチェイン修復が必要になった場合は、まず対象ツールチェインを使っていそうな
  プロセス(IDEの言語サーバ等)を確認し、ロックされそうなファイルがあれば先にリネーム/退避
  してから`uninstall`/`install`を実行する。
- システムレベルの環境変更(ツールチェイン、グローバル設定等)は、プロジェクトのコード変更
  よりも影響範囲が広い(マシン上の全プロジェクトに波及する)ため、着手前に完了までの
  手順と失敗時のリカバリ方針を明確にしてから進める。
