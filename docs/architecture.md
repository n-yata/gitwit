## アーキテクチャパターン

### レイヤードアーキテクチャ

```
┌──────────────────────────────────────┐
│   UI レイヤー（src/ui/）              │
│   egui ウィジェット・パネル描画        │
├──────────────────────────────────────┤
│   App State レイヤー（src/app.rs）    │
│   アプリ状態管理・イベントルーティング  │
├──────────────────────────────────────┤
│   Git ロジックレイヤー（src/git/）    │
│   git2 ラッパー・データ変換           │
├──────────────────────────────────────┤
│   永続化レイヤー（src/config.rs）     │
│   設定ファイルの読み書き              │
└──────────────────────────────────────┘
```

#### UI レイヤー（`src/ui/`）
- **責務**: egui の `Ui` / `Context` を使ってパネルを描画し、ユーザー操作を `AppState` に伝える
- **許可される操作**: `AppState` の読み取り・更新
- **禁止される操作**: `git2` クレートへの直接アクセス、ファイル I/O

#### App State レイヤー（`src/app.rs`）
- **責務**: アプリ全体の状態（`AppState`）を保持し、UI からのイベントを受けて Git ロジック・永続化を呼び出す
- **許可される操作**: Git ロジックレイヤーの呼び出し、永続化レイヤーの呼び出し
- **禁止される操作**: egui の `Ui` 描画 API の直接使用

#### Git ロジックレイヤー（`src/git/`）
- **責務**: `git2::Repository` をラップして、コミット一覧・diff 情報を取得する純粋な関数群
- **許可される操作**: `git2` クレートの使用・ファイル読み取り（git オブジェクト）
- **禁止される操作**: UI レイヤーへの依存、設定ファイルの読み書き

#### 永続化レイヤー（`src/config.rs`）
- **責務**: `AppConfig`（前回のリポジトリパス等）を TOML ファイルへ読み書きする
- **許可される操作**: `%APPDATA%\gitwit\` 配下へのファイル I/O
- **禁止される操作**: Git 操作・UI 描画

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| アプリ設定（前回パス等） | `%APPDATA%\gitwit\config.toml` | TOML | 人間が読めて編集しやすい |
| Git データ | Git リポジトリ本体（読み取り専用） | Git オブジェクト形式 | libgit2 が処理 |

### バックアップ戦略

設定ファイルは軽量かつ再生成可能なため、バックアップは不要。

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 起動 → コミット一覧表示 | 3 秒以内 | 500 コミット、Windows 11、SSD |
| コミット選択 → ファイル一覧表示 | 200ms 以内 | 通常規模のコミット |
| ファイル選択 → diff 表示 | 200ms 以内 | 1MB 未満のファイル |
| UI フレームレート | 60fps 維持 | 操作中 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 200MB | コミット 1,000 件 + diff キャッシュ込み |
| CPU（アイドル時） | 1% 以下 | egui は再描画のない場合 CPU を使わない |
| ディスク（アプリ本体） | 20MB 以下 | Rust バイナリ + libgit2 静的リンク |

## セキュリティアーキテクチャ

### データ保護

- **暗号化**: 不要（設定ファイルはパスのみ保存。機密情報なし）
- **アクセス制御**: `%APPDATA%\gitwit\` は OS の標準ユーザーパーミッションに委ねる
- **機密情報管理**: アプリ側でパスワード・トークン等のシークレットは一切保持・入力させない。
  `git fetch`（リモートブランチ一覧の取得、`.steering/20260718-remote-branches/`）等の認証が
  必要な操作は、すべて SSH エージェント（`ssh-agent` / `~/.ssh/`）または OS の credential helper
  （Windows 資格情報マネージャーを含む）に委譲する（`git2::RemoteCallbacks::credentials()` 経由）

### 入力検証

- **リポジトリパス**: `git2::Repository::open()` に渡して libgit2 に検証させる（パストラバーサル対策も兼ねる）
- **ファイルサイズ制限**: diff 生成前に 1MB 超を検出してスキップ
- **エラーハンドリング**: `git2::Error` を `GitError` にラップして UI 向けメッセージに変換。生のエラー詳細はログのみ

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 1,000 コミット（表示上限）
- **パフォーマンス劣化対策**: 仮想スクロール（表示範囲のウィジェットのみ生成）
- **アーカイブ戦略**: Git データはリポジトリ側で管理。アプリ側は参照のみ

### 機能拡張性

- **Post-MVP 機能追加方針**: `AppState` に新しいフィールドを追加し、新 UI パネルを `src/ui/` に追加する。既存レイヤーへの影響を最小限に保つ
- **Git ロジックレイヤーのファイル分割**: 当初は読み取り系（`read.rs`）と書き込み系（`write.rs`）で分離する方針だったが、実装時はドメイン単位（`commit.rs`, `diff.rs`, `branch.rs`, `remote.rs` 等）でファイルを分けている。ブランチ操作（一覧取得は読み取り、checkout は書き込み）のように読み書きが密接な機能は同一モジュール（`branch.rs`）にまとめる方が凝集度が高いと判断した（初のブランチ切り替え機能実装時に採用）

## 並行処理モデル

gitwit は当初、完全に同期・シングルスレッドの immediate-mode GUI として実装されていた
（`eframe::App::update()` が UIスレッド上で毎フレーム同期実行され、ローカルの git2 操作は
数ms〜数十msで完了するため体感的なフリーズがなかった）。リモートブランチ一覧の取得
（`.steering/20260718-remote-branches/`）で初めてネットワークI/O（`git fetch`、数秒かかりうる）
を導入したことに伴い、以下の並行処理モデルを標準パターンとして定める。

### 原則

- **UIスレッドは決してブロックしない**。ネットワークI/Oや長時間かかる処理は必ずバックグラウンド
  スレッドへ逃がす。
- ローカルのみで完結し数十ms以内に終わる git2 操作（コミット読み込み・diff 生成・ブランチ
  checkout 等）は、これまで通り `App::update()` 上で同期実行してよい。同期処理か非同期処理かは
  「ネットワークI/Oを伴うか」を基準に判断する。

### 標準パターン

`std::thread::spawn` + `std::sync::mpsc::channel` を使う。tokio 等の非同期ランタイムは採用しない
（git2/libgit2 は同期APIしか持たないため、非同期ランタイムを導入しても `spawn_blocking` で
スレッドに逃がすだけになり恩恵がなく、単発の非同期操作に対しては過剰設計になるため）。

1. UI 側は `state.needs_fetch = true` のような**フラグを立てるだけ**（既存の `needs_load` 等と
   同じパターン）
2. `App::update()` がフラグを検知し、`mpsc::channel()` を生成した上で `std::thread::spawn` により
   バックグラウンドスレッドを起動する
3. バックグラウンドスレッドは処理完了後、結果を `tx.send(...)` で送信し、続けて必ず
   `egui::Context::request_repaint()` を呼ぶ（**これを忘れると次のユーザー操作まで結果が画面に
   反映されない**。egui は入力がない限り再描画しないため）
4. `App::update()` の**冒頭**で `rx.try_recv()` によりポーリングし、結果を `AppState` へ反映する

### git2 のスレッド制約

`git2::Repository` は内部に libgit2 の生ポインタハンドルを持つため `Send` ではなく、スレッドを
跨いで move も参照共有もできない。バックグラウンドスレッドには `PathBuf`（`Send`）のみを渡し、
スレッド内で `GitRepository::open(path)` を新規に開き直す。UIスレッド側が保持する `App.repo`
には一切触れない。

### レイヤー分離

`egui::Context` はUI型であるため、`src/git/` レイヤーへ持ち込まない。スレッド生成・
`request_repaint()` の呼び出しは `app.rs`（ViewModel 相当のレイヤー）の責務とし、`src/git/`
配下のモジュール（例: `remote.rs`）は `&git2::Repository` を受け取る同期の純関数のみを持つ
（テスト容易性の確保と、CLAUDE.md のレイヤー構成における横断依存禁止の原則に従うため）。

## テスト戦略

### ユニットテスト
- **フレームワーク**: `cargo test`（Rust 標準）
- **対象**: `src/git/` の純粋関数（コミット変換・diff パース・日時フォーマット等）
- **カバレッジ目標**: Git ロジックレイヤーは 80% 以上

### 統合テスト
- **方法**: `tempdir` クレートで一時ディレクトリを作成し、`git init` → `git commit` でテスト用リポジトリを構築してから検証
- **対象**: `GitRepository::open()` → `load_commits()` → `load_diff_hunks()` の一連フロー

### 手動テスト（E2E）
- このプロジェクト自身のリポジトリを開いて動作確認
- 大規模リポジトリ（OSS 等）での起動・スクロール・diff 表示の速度確認

## 技術的制約

### 環境要件
- **OS**: Windows 10 / 11 のみ（egui は Windows, macOS, Linux 対応だが当面 Windows のみ）
- **必要なソフトウェア**: Git（システムにインストール済みであること。libgit2 は静的リンクするため不要）
- **最小メモリ**: 4GB 以上（推奨 8GB）
- **必要ディスク容量**: アプリ本体 20MB 以下

### パフォーマンス制約
- コミット表示上限: 1,000 件（初期実装）
- diff 表示対象: 1MB 未満のファイル（超過はスキップ）

### セキュリティ制約
- ネットワーク接続は `git fetch` など明示的なリモート操作時のみ発生する。起動・コミット履歴閲覧・
  diff 表示・ローカルブランチ切り替えはすべてネットワーク接続なしでローカル完結する
  （`.steering/20260718-remote-branches/` でリモートブランチ一覧取得のための fetch を導入）
- リモートへの書き込み（push）は行わない。リポジトリへの書き込みはローカル操作
  （ブランチ checkout 等）と、リモートからの読み取り取得である fetch のみに限定する
- 認証情報（パスワード・トークン等）はアプリで保持・入力させず、SSH エージェント / OS の
  credential helper に委譲する（詳細は「並行処理モデル」章の前、セキュリティアーキテクチャの
  機密情報管理を参照）
