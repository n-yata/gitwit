## アーキテクチャパターン

### レイヤードアーキテクチャ

```
┌──────────────────────────────────────┐
│   UI レイヤー（src/ui/）              │
│   egui ウィジェット・パネル描画        │
├──────────────────────────────────────┤
│   App State レイヤー（src/app.rs）    │
│   アプリ状態管理・イベントルーティング  │
├──────────────────────────────────────┤
│   Git ロジックレイヤー（src/git/）    │
│   git2 ラッパー・データ変換           │
├──────────────────────────────────────┤
│   永続化レイヤー（src/config.rs）     │
│   設定ファイルの読み書き              │
└──────────────────────────────────────┘
```

#### UI レイヤー（`src/ui/`）
- **責務**: egui の `Ui` / `Context` を使ってパネルを描画し、ユーザー操作を `AppState` に伝える
- **許可される操作**: `AppState` の読み取り・更新
- **禁止される操作**: `git2` クレートへの直接アクセス、ファイル I/O

#### App State レイヤー（`src/app.rs`）
- **責務**: アプリ全体の状態（`AppState`）を保持し、UI からのイベントを受けて Git ロジック・永続化を呼び出す
- **許可される操作**: Git ロジックレイヤーの呼び出し、永続化レイヤーの呼び出し
- **禁止される操作**: egui の `Ui` 描画 API の直接使用

#### Git ロジックレイヤー（`src/git/`）
- **責務**: `git2::Repository` をラップして、コミット一覧・diff 情報を取得する純粋な関数群
- **許可される操作**: `git2` クレートの使用・ファイル読み取り（git オブジェクト）
- **禁止される操作**: UI レイヤーへの依存、設定ファイルの読み書き

#### 永続化レイヤー（`src/config.rs`）
- **責務**: `AppConfig`（前回のリポジトリパス等）を TOML ファイルへ読み書きする
- **許可される操作**: `%APPDATA%\gitwit\` 配下へのファイル I/O
- **禁止される操作**: Git 操作・UI 描画

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| アプリ設定（前回パス等） | `%APPDATA%\gitwit\config.toml` | TOML | 人間が読めて編集しやすい |
| Git データ | Git リポジトリ本体（読み取り専用） | Git オブジェクト形式 | libgit2 が処理 |

### バックアップ戦略

設定ファイルは軽量かつ再生成可能なため、バックアップは不要。

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 起動 → コミット一覧表示 | 3 秒以内 | 500 コミット、Windows 11、SSD |
| コミット選択 → ファイル一覧表示 | 200ms 以内 | 通常規模のコミット |
| ファイル選択 → diff 表示 | 200ms 以内 | 1MB 未満のファイル |
| UI フレームレート | 60fps 維持 | 操作中 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 200MB | コミット 1,000 件 + diff キャッシュ込み |
| CPU（アイドル時） | 1% 以下 | egui は再描画のない場合 CPU を使わない |
| ディスク（アプリ本体） | 20MB 以下 | Rust バイナリ + libgit2 静的リンク |

## セキュリティアーキテクチャ

### データ保護

- **暗号化**: 不要（設定ファイルはパスのみ保存。機密情報なし）
- **アクセス制御**: `%APPDATA%\gitwit\` は OS の標準ユーザーパーミッションに委ねる
- **機密情報管理**: 認証情報・SSH キーは扱わない（MVP は読み取り専用でリモートアクセスなし）

### 入力検証

- **リポジトリパス**: `git2::Repository::open()` に渡して libgit2 に検証させる（パストラバーサル対策も兼ねる）
- **ファイルサイズ制限**: diff 生成前に 1MB 超を検出してスキップ
- **エラーハンドリング**: `git2::Error` を `GitError` にラップして UI 向けメッセージに変換。生のエラー詳細はログのみ

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 1,000 コミット（表示上限）
- **パフォーマンス劣化対策**: 仮想スクロール（表示範囲のウィジェットのみ生成）
- **アーカイブ戦略**: Git データはリポジトリ側で管理。アプリ側は参照のみ

### 機能拡張性

- **Post-MVP 機能追加方針**: `AppState` に新しいフィールドを追加し、新 UI パネルを `src/ui/` に追加する。既存レイヤーへの影響を最小限に保つ
- **書き込み系機能（ステージング・コミット等）**: `src/git/write.rs` として Git ロジックレイヤーに追加し、読み取り系（`src/git/read.rs`）とファイルを分離する

## テスト戦略

### ユニットテスト
- **フレームワーク**: `cargo test`（Rust 標準）
- **対象**: `src/git/` の純粋関数（コミット変換・diff パース・日時フォーマット等）
- **カバレッジ目標**: Git ロジックレイヤーは 80% 以上

### 統合テスト
- **方法**: `tempdir` クレートで一時ディレクトリを作成し、`git init` → `git commit` でテスト用リポジトリを構築してから検証
- **対象**: `GitRepository::open()` → `load_commits()` → `load_diff_hunks()` の一連フロー

### 手動テスト（E2E）
- このプロジェクト自身のリポジトリを開いて動作確認
- 大規模リポジトリ（OSS 等）での起動・スクロール・diff 表示の速度確認

## 技術的制約

### 環境要件
- **OS**: Windows 10 / 11 のみ（egui は Windows, macOS, Linux 対応だが当面 Windows のみ）
- **必要なソフトウェア**: Git（システムにインストール済みであること。libgit2 は静的リンクするため不要）
- **最小メモリ**: 4GB 以上（推奨 8GB）
- **必要ディスク容量**: アプリ本体 20MB 以下

### パフォーマンス制約
- コミット表示上限: 1,000 件（初期実装）
- diff 表示対象: 1MB 未満のファイル（超過はスキップ）

### セキュリティ制約
- MVP はネットワーク接続なし（リモートアクセスなし）
- リポジトリへの書き込みは行わない
